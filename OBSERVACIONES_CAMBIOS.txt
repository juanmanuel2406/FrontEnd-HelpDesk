----------------------------------------
 ERRORES ORIGINALES DETECTADOS
----------------------------------------

1. BACKEND - TicketCreateDTO:
   - El campo Priority era string no nullable (string). Si el frontend
     enviaba "priority": null, System.Text.Json lanzaba una excepción
     de serialización antes de llegar al controlador.
   - Faltaban atributos [Required] y [MaxLength] en Title y Description
     para validación en nivel de modelo.
   - El DTO de actualización (TicketUpdateDTO) tenía todos los campos
     como obligatorios, impediendo actualizaciones parciales.

2. BACKEND - TicketController:
   - Ningún endpoint tenía try-catch. Cualquier excepción no controlada
     devolvía un error 500 genérico sin detalles.

3. BACKEND - TicketService:
   - El campo Status no se seteaba explícitamente (dependía del default
     de la entidad).
   - Los mensajes de error estaban en inglés.

4. BACKEND - SQLite / Foreign Keys:
   - La conexión a SQLite no tenía Foreign Keys=True, por lo que las
     restricciones de clave foránea no se validaban.

5. BACKEND - Program.cs:
   - No se suprimía el filtro automático de validación de modelos
     (SuppressModelStateInvalidFilter), causando que errores de
     validación devolvieran un formato distinto al GeneralResponse.

6. BACKEND - Modelo LoginResponse:
   - Faltaban las propiedades Fullname y Email, por lo que el frontend
     recibía undefined al intentar mostrar el perfil del usuario.

7. FRONTEND - Ausencia de AuthInterceptor:
   - No existía un HttpInterceptor que inyectara automáticamente el
     token JWT en las peticiones. Cada método del servicio recibía
     el token como parámetro manual, y si el llamador no lo pasaba,
     la request llegaba sin autenticación al backend (Error 401).

8. FRONTEND - TicketService:
   - Todos los métodos requerían el parámetro token manualmente,
     generando código repetitivo y propenso a errores.

9. FRONTEND - app.ts:
   - El getter token() y el pase manual de this.token a cada llamada
     del servicio era frágil. Si el token expiraba o no estaba
     disponible, la request fallaba con 401.
   - Los callbacks de error no mostraban el mensaje del backend.

========================================
 ANÁLISIS DEL ERROR 401 (JWT)
========================================

Se verificó el flujo completo de autenticación JWT:

1. GENERACIÓN (JwtHandler.CrearJWT):
   - Usa IConfiguration para leer Secret, Issuer, Audience
   - Agrega ClaimTypes.NameIdentifier con el userId como string
   - Agrega ClaimTypes.Role con el rol del usuario
   - Usa HMAC-SHA256 con la clave de appsettings.json

2. VALIDACIÓN (Program.cs - AddJwtBearer):
   - ValidateIssuer = true con ValidIssuer = "api.helpdesk"
   - ValidateAudience = true con ValidAudience = "helpdesk"
   - ValidateLifetime = true
   - ValidateIssuerSigningKey = true
   - Misma clave que en generación

3. CONCLUSIÓN:
   - La generación y validación del JWT en el backend es CORRECTA
   - El error 401 ocurría porque el frontend NO enviaba el token
     en el header Authorization
   - Solución: creación del AuthInterceptor que inyecta el token
     automáticamente en todas las peticiones
